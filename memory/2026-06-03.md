# 2026-06-03

## Bug fixed: MCP timeout too short for large drawings

Opened a real gen-plan ("სამგორი ბ") with **72,561 entities** (BlockReference 21,317;
Polyline 28,543; Hatch 5,508; Line 7,459; RotatedDimension 3,740; Arc 2,961; MText 1,954;
Wipeout 531; Leader 493; Circle 42; RasterImage 5).

Symptom: `count_entities`, `query_entities`, `list_blocks`, `get_drawing_extents` all
returned **error 500** via the MCP tools, while `get_layers` and `run_command` worked.

Diagnosis: posting directly to the plugin (`curl http://localhost:12345/` with the Bearer
token, body `{"Command":"count_entities","Args":{}}`) returned **HTTP 200** with the full
count. So the plugin was fine — the bug was in the MCP server.

REAL root cause (the 5s timeout was a red herring — the plugin answers in <1s):
`src/index.ts` `pluginCommandMap` maps most tool names to **PascalCase** method names
(e.g. `count_entities: "CountEntities"`) and forwarded that to the plugin. But the plugin's
`CommandProcessor.Process` does `request.Command.ToLower()` and switches on **snake_case**
cases (`"count_entities"`). So `"CountEntities".ToLower()` = `"countentities"` never matches
→ `throw new ArgumentException("Unknown command: CountEntities")` → HTTP 500.
Verified directly: `curl ... -d '{"command":"CountEntities",...}'` → 500 "Unknown command";
`-d '{"command":"count_entities",...}'` → 200. Only the few tools left as snake_case in the
map (`get_layers`, `run_command`, `create_line`, `create_layer`, `insert_block`) worked.

Fix: in `src/index.ts`, forward the original snake_case tool `name` instead of the mapped
PascalCase value: `return pluginForward(name, args);`. The map stays as an allow-list.
Also kept the `autocadClient.ts` timeout bump 5s→120s (`AUTOCAD_PLUGIN_TIMEOUT_MS`) as a
safety margin for genuinely large traversals.

Rebuilt with `npm run build` (needs `NODE_OPTIONS=--use-system-ca`).
**Requires restarting Claude** so the MCP server reloads `dist/index.js` — the running
server keeps the old code until then.

## Debugging trick worth keeping
To see the plugin's *real* error (the MCP server only surfaces "status code 500"), POST
straight to the plugin:
`curl -s -X POST http://localhost:12345/ -H "Authorization: Bearer default-secret-token" -H "Content-Type: application/json" -d '{"Command":"<cmd>","Args":{}}'`
